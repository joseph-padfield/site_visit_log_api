# Site Visit Log API

A FastAPI service for a fictional heritage consultant, tracking `Site` records (buildings or places under observation) and `Visit` records (dated inspections tied to exactly one site). Built as a learning exercise practising PostgreSQL, Docker Compose, environment-based configuration, SQLAlchemy's declarative syntax, one-to-many relationships with foreign keys, and Alembic migrations from the first commit.

## Status

Complete. All three parts of the build (setup/Docker/configuration, models/migrations/routes, and testing) are finished: the `Site`/`Visit` schema is modelled and migrated across two Alembic revisions, all seven endpoints are implemented, and the pytest suite covers site creation, unknown-site 404s, nested visit creation, unknown-parent 404s, and the `since` date filter, running against a separate `site_visit_log_test` database.

## Tech stack

- FastAPI
- PostgreSQL 17, via Docker Compose
- SQLAlchemy 2.0 (declarative style)
- Alembic (schema migrations)
- pytest + httpx (endpoint testing, via `TestClient` and dependency overrides)

## Data model

`Site 1 ───< Visit`, with `visits.site_id` a foreign key to `sites.id` (`ON DELETE CASCADE`).

| Table | Columns |
|---|---|
| `sites` | `id`, `name`, `address`, `site_type`, `notes`, `created_at` |
| `visits` | `id`, `site_id` (FK), `visit_date` (indexed), `purpose`, `weather`, `notes`, `follow_up_required` |

`follow_up_required` was added in a second migration after the initial schema, as a non-nullable boolean defaulting to `false`.

Update and delete routes are intentionally out of scope — the exercise is about relationships, migrations and configuration, not full CRUD.

## Setup

Clone the repo and create a virtual environment:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Copy the environment file and fill in real values:

```bash
cp .env.example .env
```

`.env` needs a `DATABASE_URL` and `TEST_DATABASE_URL`, matching the credentials in `compose.yaml`:

```
DATABASE_URL=postgresql+psycopg://<user>:<password>@localhost:5432/<database_name>
TEST_DATABASE_URL=postgresql+psycopg://<user>:<password>@localhost:5432/<test_database_name>
```

Start PostgreSQL:

```bash
docker compose up -d
docker compose ps   # confirm the db service is healthy
```

Apply migrations to the development database:

```bash
alembic upgrade head
```

Confirm the schema, if you want to check directly:

```bash
docker compose exec db psql -U postgres -d site_visit_log
\dt
\d sites
\d visits
```

At the end of a session:

```bash
docker compose down
```

## Running the app

```bash
uvicorn app.main:app --reload
```

The API is then at `http://127.0.0.1:8000`, with interactive docs (Swagger UI) at `http://127.0.0.1:8000/docs`.

## Database migrations

Every schema change goes through Alembic, never a manual `ALTER` or a dropped/recreated table.

```bash
alembic revision --autogenerate -m "describe the change"   # draft a migration
alembic upgrade head                                        # apply pending migrations
alembic downgrade -1                                         # reverse the last one
alembic current                                              # show the current revision
alembic history                                               # show the full chain
```

Always review an autogenerated migration against `app/models.py` before applying it — autogenerate drafts a diff, it does not guarantee correctness.

## Testing

Tests run against a separate `site_visit_log_test` database, never the development one.

Create it (once) inside the same container:

```bash
docker compose exec db createdb -U postgres site_visit_log_test
```

Apply migrations to it, overriding `DATABASE_URL` for the command only:

```bash
DATABASE_URL=postgresql+psycopg://postgres:postgres@localhost:5432/site_visit_log_test \
alembic upgrade head
```

Run the suite:

```bash
pytest
pytest -v   # more detail
```

The `client` fixture in `tests/conftest.py` overrides FastAPI's `get_db` dependency with a session bound to the test database, and an autouse `clean_database` fixture truncates `visits`/`sites` (resetting identities) before and after every test, so each test starts from an empty, predictable state.

## Endpoints

- `POST /sites/`
- `GET /sites/`
- `GET /sites/{site_id}`
- `POST /sites/{site_id}/visits/`
- `GET /sites/{site_id}/visits/`
- `GET /visits/`
- `GET /visits/?since=2026-01-01`

Unknown sites return `404` on every route that depends on one (`GET /sites/{site_id}`, and both `POST`/`GET /sites/{site_id}/visits/`). The `since` filter on `GET /visits/` is inclusive (`visit_date >= since`) and results are ordered newest first.

## Out of scope

- Update and delete routes for `Site`/`Visit`
- Authentication
- Many-to-many relationships
- Asynchronous database access
- Deployment (local Docker Compose only)
